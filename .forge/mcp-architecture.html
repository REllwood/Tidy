Architecture Recommendation · Research

Hosting an MCP server inside AppFlower

A firm plan for letting Claude Code and Codex push meeting notes into the vault, read and search the knowledge base, and manage tasks & tables — grounded in the app's existing Tauri / Rust core and rusqlite store.

Base Tauri 2 · React 19 · Rust core Store rusqlite (WAL, FTS5) Spec basis MCP 2025-11-25 Dated July 2026
The recommendation, in one breath

Ship a standalone Rust MCP binary built on rmcp 2.1, bundled as a Tauri sidecar, speaking stdio to same-machine AI clients. It opens the same appflower.db in WAL mode and reuses your existing core:: functions — no HTTP port, no network surface, no second language runtime.

01

Transport: stdio, not HTTP

The current spec (protocol revision 2025-11-25) defines exactly two standard transports: stdio and Streamable HTTP. The old HTTP+SSE transport is deprecated — retained only for backwards compatibility and losing client support over time. Do not build a new SSE server.

Use this

stdio

The client launches the server as a subprocess and exchanges newline-delimited JSON-RPC over stdin/stdout. No port, no listening socket, no HTTP surface an attacker can reach.

The spec: "Clients SHOULD support stdio whenever possible." Both Claude Code and Codex do, and both make it the default for local servers.

Not needed

Streamable HTTP

Right for remote, multi-client, cloud-hosted servers. For a desktop app whose only clients are CLIs on the same machine, it adds a bound socket, an Origin-validation obligation, and a token handshake — cost with no benefit.

Why stdio wins here specifically. Your clients are Claude Code and Codex running on the user's own Mac. A subprocess pipe is the tightest possible trust boundary: only a local process that can already spawn the binary can talk to it. Streamable HTTP's own security section exists precisely to recover that property for network servers — it MUST validate the Origin header against DNS-rebinding, SHOULD bind only to 127.0.0.1, and SHOULD authenticate every connection. With stdio you get all of that for free by construction.

Keep the door open

rmcp makes the transport a one-line swap. If you later want a remote client (a phone, a hosted agent), add the transport-streamable-http-server feature and bind to 127.0.0.1 behind a bearer token — the tool code doesn't change. Ship stdio now; don't foreclose HTTP.

02

SDK & runtime: rmcp, as a sidecar binary

Two decisions here: which SDK, and where the server process lives relative to the Tauri app. Both point the same way — stay in Rust, stay close to the store.

Why rmcp over shipping a Node server

  • It's the official Rust SDK (the rmcp crate, modelcontextprotocol/rust-sdk). Current version 2.1.0 (2026-07-02); the wider ecosystem tracks the latest protocol revision.
  • One language, one store. Your core is already Rust with rusqlite. The MCP server can call the exact same commands::pages::core, databases::core, and search::core functions the Tauri commands call. A Node server would re-implement every KB operation in JS/TS against the same SQLite file — duplicated logic, drift risk, and a second runtime to bundle and code-sign.
  • Ergonomic tool definitions. The #[tool_router] / #[tool] / #[tool_handler] macros generate the JSON-Schema and dispatch from your function signatures. Tool inputs arrive as typed Parameters<T> structs.
  • You already depend on tokio, serde, reqwest. rmcp is a natural fit — no new heavy toolchain.

In-process vs. sidecar vs. standalone

No

In-process (inside Tauri)

The AI client needs to spawn its MCP server and own its stdin/stdout. It can't attach to your already-running GUI process's pipes. stdio fundamentally requires a spawnable executable, so an in-process server can't serve external CLIs.

Ship this

Sidecar binary

A second Rust binary in the same Cargo workspace, bundled via Tauri's externalBin. Ships and code-signs inside the .app, versioned in lockstep with the core, sharing the appflower_lib crate. The user points their AI tool at its absolute path.

A pure standalone binary (published separately, installed by the user) is the same code but a worse distribution story — version skew against the app, a separate install step, no shared code-signing. The sidecar is a bundled standalone binary; let Tauri ship it.

The one hard constraint — WAL, one writer

Your core comment says the Rust core is "the sole writer." The sidecar is now a second process opening the same DB file. SQLite WAL mode supports exactly one writer at a time across processes, coordinated by the OS. Open the sidecar's connection with PRAGMA journal_mode=WAL and, critically, PRAGMA busy_timeout=5000 so a concurrent write from the GUI doesn't return SQLITE_BUSY — it waits. Every FTS5 trigger and migration still runs; the sidecar must run migrations to the same version on open (or refuse to start if the app's schema is newer).

03

The sidecar, concretely

A minimal but real skeleton. The server struct holds a DB handle; each #[tool] is a thin wrapper over an existing core:: function.

Cargo.toml — new bin crate: appflower-mcprmcp 2.1
[dependencies]
rmcp = { version = "2.1", features = [
  "server", # default: tool system + ServerHandler
  "macros", # default: #[tool] / #[tool_router]
  "transport-io" # server-side stdio transport
] }
appflower_lib = { path = "../src-tauri" }  # reuse core::, db::
tokio = { version = "1", features = ["full"] }
rusqlite = { version = "0.32", features = ["bundled"] }
serde = { version = "1", features = ["derive"] }
main.rs — tools wrap existing core functionsRust
use rmcp::{
    handler::server::{router::tool::ToolRouter, wrapper::Parameters},
    model::*, tool, tool_handler, tool_router,
    ServiceExt, ServerHandler, ErrorData as McpError,
    transport::stdio,
};
use appflower_lib::{db::Db, commands::{pages, search, databases}};

#[derive(Clone)]
struct Kb { db: std::sync::Arc<Db> }

#[derive(serde::Deserialize, schemars::JsonSchema)]
struct SearchArgs { /// full-text query
    query: String }

#[tool_router]
impl Kb {
    #[tool(description = "Full-text search the knowledge base (pages + table rows)")]
    async fn kb_search(&self, Parameters(a): Parameters<SearchArgs>)
        -> Result<CallToolResult, McpError>
    {
        let conn = self.db.conn.lock().unwrap();
        let hits = search::core::search(&conn, &a.query)   // ← existing code
            .map_err(|e| McpError::internal_error(e.to_string(), None))?;
        Ok(CallToolResult::success(vec![Content::json(hits)?]))
    }
    // ... ingest_note, create_page, add_row, set_cell, etc.
}

#[tool_handler]
impl ServerHandler for Kb {
    fn get_info(&self) -> ServerInfo {
        ServerInfo {
            capabilities: ServerCapabilities::builder().enable_tools().build(),
            instructions: Some("AppFlower knowledge base: notes, tasks, tables.".into()),
            ..Default::default()
        }
    }
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    // Open the SAME db the app uses; WAL + busy_timeout for cross-process writes.
    let db = Db::open(&appflower_db_path())?;   // runs migrations, sets pragmas
    let service = Kb { db: db.into() }.serve(stdio()).await?;
    service.waiting().await?;                    // stderr only for logs — never stdout
    Ok(())
}

Note the discipline stdio demands: nothing but valid JSON-RPC may touch stdout. Route all logging to stderr (as the app's tauri-plugin-log already models).

04

Registering with Claude Code & Codex

Both tools register a local stdio server as "run this command." The user (or a one-click helper you build into AppFlower's settings) points them at the bundled sidecar's absolute path.

Claude Code

Add via CLI — the -- separates Claude's flags from the command it will run:

terminalclaude mcp add
# user scope = available in every project on this machine
claude mcp add --scope user --transport stdio appflower \
  -- "/Applications/AppFlower.app/Contents/MacOS/appflower-mcp"

Scopes: local (default, this project only, in ~/.claude.json) · user (all your projects) · project (shared, committed to .mcp.json). For a personal desktop app the KB belongs to one user, so user scope is the right default. The equivalent .mcp.json / ~/.claude.json entry:

~/.claude.json (or .mcp.json for a shared project)JSON
{
  "mcpServers": {
    "appflower": {
      "type": "stdio",
      "command": "/Applications/AppFlower.app/Contents/MacOS/appflower-mcp",
      "args": [],
      "env": { "APPFLOWER_MCP_TOKEN": "${APPFLOWER_MCP_TOKEN}" }
    }
  }
}

Codex

Codex reads ~/.codex/config.toml (or a project-scoped .codex/config.toml). Each server is a [mcp_servers.<name>] table; a stdio server needs command + optional args / env:

~/.codex/config.tomlTOML
[mcp_servers.appflower]
command = "/Applications/AppFlower.app/Contents/MacOS/appflower-mcp"
args = []

[mcp_servers.appflower.env]
APPFLOWER_MCP_TOKEN = "…"   # handshake secret, see §5

Or via the CLI: codex mcp add appflower -- /Applications/AppFlower.app/Contents/MacOS/appflower-mcp.

One-click onboarding

Don't make the user hand-type an .app path. AppFlower's settings screen should offer "Connect Claude Code" / "Connect Codex" buttons that shell out to claude mcp add … / write the config.toml table with the correct resolved sidecar path and a freshly minted token — the same UX pattern first-party MCP integrations ship.

05

Security posture for a KB the AI can write to

A read-only server is low-stakes. This one creates notes, edits tables, and deletes rows — a prompt-injected agent could do real damage. The 2025-11-25 security guidance and defense-in-depth both apply, even on stdio.

  • No network surface by default. stdio means there is no socket to bind, no Origin to validate, no remote reachability. This is the single biggest security win over an HTTP server and the reason to prefer stdio. If you ever add HTTP: bind 127.0.0.1 only, validate Origin (spec: MUST, 403 on mismatch), require a bearer token.
  • Handshake token even on stdio. Any local process that knows the path can spawn the sidecar. Have the app write a random per-install secret into a setting; the sidecar reads it from its own DB and requires the client to present it (via the env shown above, checked at initialize). This stops other local software from silently registering itself against your KB.
  • Scope tools to intent, gate destructive ones. Split read tools (kb_search, get_page) from write tools (ingest_note, delete_row). Consider a config flag that ships read-only by default, with writes opt-in. Deletes should be soft (archive), not hard, so an agent mistake is recoverable.
  • Validate at the boundary. Tool inputs are untrusted model output. Reuse the app's own guards — e.g. search::core already escapes FTS/LIKE wildcards and quotes tokens; keep every write path going through core:: so field-type and page-type validation ('doc'|'database') is never bypassed.
  • Explicit user consent to run a local binary. The spec calls this out for local servers: the client shows the exact command and the user approves execution. Claude Code's project-scope approval prompt does exactly this — lean into it; don't try to auto-register silently.
  • No secrets in tool text. Never surface the handshake token, file paths, or audio paths back through tool results beyond what the task needs.

Proposed tool surface

Three capability groups, mapped onto your existing entities (page, database, field, db_row, cell, meeting) and reusing your core:: functions. read write flagship

ToolParamsDoes
A · Read & search the KB
kb_search query FTS5 over page title+content and text cells. Wraps search::core::search. read
list_pages parent_id? Tree of pages (doc + database), for navigation. read
get_page page_id, format? Page metadata + BlockNote content, optionally rendered to markdown. read
get_database page_id Fields, rows, cells, and views for a database page. read
B · Manage tasks & tables
create_page title, kind, parent_id? New doc or database page. Wraps pages::core::create (validates kind). write
update_document page_id, content|markdown Replace/append BlockNote body; FTS index updates via trigger. write
create_field database_id, name, type, options? Add a column (text|number|select|date|checkbox). write
add_row database_id, cells? Insert a row, optionally with initial cell values. write
set_cell row_id, field_id, value Set one typed cell. Wraps databases::core::set_cell. write
archive_row row_id Soft-delete (recoverable) rather than hard delete_row. write
C · Ingest meeting notes — the flagship
ingest_note raw_text, client_hint?, meeting_id?, task_db_id? Parse raw meeting text → create/update a note page → link the client entity → extract action items into task rows. See §7. flagship
Resources, not just tools

MCP also has resources (read-only, addressable context the client can pull in). Expose pages as resources under an appflower://page/{id} URI scheme so an agent can reference a note as context without a tool round-trip. Keep writes as tools (they need arguments and confirmation); keep reads available as both a tool (for search) and resources (for direct reference).

07

Flagship flow: ingest_note end-to-end

The AI hands over raw meeting text; the server does the filing. This is the highest-value path and the one to build first.

  • 1 · Parse. Split the raw text into summary, decisions, and action items. You already have this shape: llm::ollama::summarize_transcript returns exactly {summary, action_items[], decisions[]}. Reuse it — the local Ollama does the extraction, the MCP tool orchestrates the filing.
  • 2 · Match the client entity. Use client_hint (or the parsed attendee list) to kb_search the "Clients" database for an existing row; create one if none matches. This is the "link the right client" step.
  • 3 · Write the note page. create_page (kind doc) under the client, body = the BlockNote-rendered summary + decisions. FTS5 indexes it automatically via the existing page_fts_ai trigger.
  • 4 · Link the meeting. If meeting_id is supplied, set its page_id so the recording, transcript, and note are joined — exactly what record_meeting already models.
  • 5–6 · Extract → tasks. Each action item becomes a db_row in the tasks database, with cells for title, status (select), and a link back to the client. One add_row per item, all in a single SQLite transaction so a failure leaves no half-filed note.
Return value

Reply with the created page_id, the resolved/created client_row_id, and the list of task row_ids — so the agent can confirm ("Filed under Acme Corp, 3 tasks created") and deep-link the user into the app. Wrap the whole thing in one transaction; surface a structured error, never a partial write.

08

Bundling it as a sidecar

Tauri ships the binary inside the .app and code-signs it with everything else.

src-tauri/tauri.conf.json — bundle blockJSON
"bundle": {
  "externalBin": ["binaries/appflower-mcp"]
  // requires binaries/appflower-mcp-aarch64-apple-darwin (+ x86_64 variant)
  // built from the appflower-mcp crate for each target triple
}
  • The suffixed target-triple name (-aarch64-apple-darwin) is how Tauri picks the right binary per arch — mirror your existing whisper Metal/CPU split.
  • The app itself need not spawn it (the AI client does), so you likely don't need the shell-plugin sidecar() permission — but keep the option for a future "test connection" button in settings.
  • Resolve the installed path once at first run and feed it to the one-click claude mcp add / config.toml writer.
